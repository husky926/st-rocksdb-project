# 14.29M 键 PKDD 实验：给非数据库方向读者的说明 + 我们的设计改了什么

本文面向**不熟悉 RocksDB / LSM / 轨迹数据库**的读者，用尽量直白的话说明：**这次实验在比什么**、**窗口是什么**、**B0 / A / full_scan / prune** 各指什么，以及**我们在 Manifest、SST、读路径上做了哪些改造**、**为何能测到明显差距**、**能否说明设计有效**。

---

## 第一部分：背景——我们在解决什么问题

### 1.1 数据长什么样

- 我们把 **轨迹数据**（例如出租车 GPS 点）存进 **RocksDB** 里：每个点对应 **一条键值对（KV）**。
- 键里带有该点的 **时间** 和 **经纬度**（以及我们自己的编码格式），值是点的其它载荷。
- 一次「查询」在这里抽象成：给定一个 **时空范围**——  
  **时间区间** \([t_{min}, t_{max}]\)，以及地面上的一个 **矩形范围**（经度区间 × 纬度区间）——  
  希望 **枚举出落在这个盒子里的所有点**（在工程上等价于用迭代器按某种顺序访问这些 KV）。

### 1.2 RocksDB 一行字

**RocksDB** 是一种 **键值存储**：数据按 **键的字典序** 落在多层 **SST 文件**里，由 **Manifest** 记录「当前有哪些文件、在几层、键的大致范围」。读数据时要 **按 LSM 的正常路径** 打开文件、读索引块、再读数据块——**不是**「不查索引把硬盘从头到尾读一遍」那种粗糙意义上的「全盘 raw 读」。

### 1.3 「列族（CF）」是什么

可以粗浅理解成：**同一张逻辑表里的所有轨迹点**放在 RocksDB 的一个 **Column Family** 里。我们实验里的 **「全表扫」** 指的是：**在这个列族里，从最小键一直迭代到最大键**，把 **这一整张表里的每一个点都路过一遍**（在基线模式下，还会把每个点的值都按迭代语义读出来）。

## 第二部分：实验在比什么——B0、A、full_scan、prune、read_bench

下面这些代号都是为了 **同一套对比** 服务的，只是 **粒度不同**（论文里叫 B0/A，工具日志里叫 full_scan/prune_scan）。

### 2.1 B0（基线）是什么

- **B0** = **关闭时空剪枝**时的读路径。
- 技术上：`ReadOptions` 里 **`experimental_st_prune_scan.enable = false`**。
- 行为：对目标列族做 **`SeekToFirst`（到第一个键）**，然后不断 **`Next`**，直到没有下一个键——即 **整列族从头到尾扫一遍**。
- **重要**：在这种模式下，**查询窗口（时间 + 经纬度盒子）不参与决策**。无论你问的是「全地球一小时」还是「东海里一个 1m² 盒子」，**扫表的路径和读盘量（在我们统计的语义下）都一样**——因为实现上就是 **「把整个 CF 迭代完」**。

**直觉**：B0 回答的是——**如果引擎完全不知道「我只要窗里的点」、只能老实把整个表走一遍，要花多少时间、读多少字节。**

### 2.2 A（剪枝路径）是什么

- **A** = **打开时空剪枝**后的同一条 **「从前向后迭代」** API（仍然是 `SeekToFirst` + `Next`），但内部会利用 **文件级 / 块级 / 键级** 的时空元数据，**跳过**与查询窗 **不相交** 的文件、块或键。
- 技术上：`experimental_st_prune_scan.enable = true`，并传入 **与本次查询对应的** \(t\)、\(x\)、\(y\) 范围。

**直觉**：A 回答的是——**在同样的迭代接口下，若引擎知道查询窗，能否少打开文件、少读块、少枚举键。**

### 2.3 `full_scan` 和 `prune_scan`（工具打印的名字）

这是我们自研小程序 **`st_meta_read_bench`**（在 `rocks-demo` 里）对上面两种模式的 **日志标签**：

| 日志里的名字 | 对应 | 含义 |
|--------------|------|------|
| **`full_scan`** | B0 | 一次 **无剪枝** 的整 CF 迭代；打印 **键数、块读计数、墙钟时间、IO 统计里的 bytes_read** 等。 |
| **`prune_scan`** | A | **同一进程、同一库、同一套 bench**，再跑一遍 **带查询窗的剪枝迭代**；同样打印上述指标。 |

两者都用 **同一个可执行文件、同一套 RocksDB fork**，差别 **仅在于** 是否开启 `experimental_st_prune_scan` 以及是否传入窗参数。

### 2.4 `st_meta_read_bench` 是什么（read_bench）

- **`st_meta_read_bench`**：**打开已经灌好的数据库目录**，**不建库、不删数据**，专门用来量 **真实读路径** 上的：
  - **墙钟时间**（`wall_us`）；
  - **从存储读了多少字节**（`IOStatsContext` 里的 **`bytes_read`**，我们报告的是该轮前后的 **增量**）；
  - **迭代器语义下数了多少个键**（full 下是全表键数；prune 下是 **剪枝后仍暴露给你的键数**，与「窗内点」对齐的实现目标一致）。
- 它不是「抽象模型」，而是 **真打开 DB、真走 BlockBasedTable / Version / Iterator**。

### 2.5 我们最常引用的两个比值

- **`ratio_bytes = prune_bytes_read / full_bytes_read`**  
  **剪枝路径**相对 **无剪枝全表迭代**，**少读了多少比例的数据**（从 OS/存储视角的字节计数）。**越小说明读盘越省。**

- **`ratio_wall = prune_wall_us / full_wall_us`**  
  **剪枝路径**用了全表路径 **多少比例的墙钟时间**。通常与 `ratio_bytes` **同向**，但不一定相等（CPU、索引路径、缓存命中等都会影响）。

另外还有 **`key_selectivity = prune_keys / full_keys`**：**枚举到的键**占 **全表键数**的比例，反映 **逻辑上** 查询有多「挑」。

## 第三部分：「查询窗口」到底是什么——本次实验里两类窗

查询窗口 = **用户声明的一个轴对齐的时空盒子**（时间一段 + 地面矩形）。  
**不相交**的几何直觉：若某个文件（或块）里所有点的 \((t,x,y)\) 都 **整体落在盒子外**，就可以 **整块跳过**，不必读数据块内容。

### 3.1 设计好的 12 个窗口（`st_validity_experiment_windows_pkdd.csv`）

这是 **人为设计** 的一组代表场景（名字是英文标签，下面是**用途白话**）：

| 标签（简写） | 大致意图（白话） |
|--------------|------------------|
| **wide_baseline** | **大窗口**：时间跨度长、空间覆盖 Porto 核心条带——**很多点会落在窗里**，用来代表「肥查询」。 |
| **sharp_spatiotemporal** | **又窄时间又小空间**——**很挑**的点查询。 |
| **east_x_file_disjoint** | 经度取在 **数据整体范围以东**（海里/陆外）——期望 **整个数据文件与窗在文件级就不相交**，剪枝极强。 |
| **west_x_both_intersect** | 偏西的条带，仍与数据有交集。 |
| **tight_y_band** | 纬度很窄的一条带。 |
| **early_t_slice / mid_t_slice / late_t_slice** | 只扫 **时间轴上早/中/晚** 的一段，空间仍用核心条带。 |
| **south_y_shift / north_y_shift** | 空间整体 **偏南 / 偏北**，命中密度不同。 |
| **x_near_eastern_boundary** | 靠近数据 **东侧经度边界** 的条带。 |
| **micro_box** | **很小**的时空盒子。 |

**14.29M 实验**里，对这 12 个窗各跑了一次 **full + prune**，结果在：  
`data/experiments/prune_vs_full_pkdd_large_design_windows.tsv`。

### 3.2 随机窗口 + 锚点

- 在 **数据整体包络**（时间 + 经纬度的外接矩形）里 **均匀随机** 抽 **轴对齐盒子**（有最小时空跨度、有最大跨度比例上限，避免退化）。
- **锚点 `wide_baseline_anchor`**：与上面 **wide_baseline** 同款几何，**不是随机的**，用来和「手设计宽窗」对齐。
- 大规模库用的是 **`--envelope pkdd_large_300k`**，与 **30 万段轨迹子集** 的实际范围一致。

随机窗的 TSV 示例：  
`data/experiments/prune_vs_full_pkdd_large_random_n25_s43.tsv`（25 随机 + 1 锚点，seed=43）。

### 3.3 图在哪里

浏览器打开 **`docs/st_prune_vs_full_pkdd_large_charts.html`**：设计窗的 bytes/wall 比、随机窗分布都有条形图示意。

## 第四部分：本次 14.29M 实验「数据」具体指什么

| 项 | 内容 |
|----|------|
| **数据来源** | PKDD / Porto 出租车竞赛 **train.csv.zip**，用 `tools/process_pkdd_train.py` 抽出 **前 30 万条有效轨迹**，展开成 **点 CSV**（`data/processed/pkdd_large`）。 |
| **点数** | 约 **14,288,269** 个点 → 约 **14,288,269** 条 ST 用户键。 |
| **库目录** | `data/verify_pkdd_st_large`（**空目录起灌**，`st_meta_smoke --csv ... --st-keys`）。 |
| **全表迭代（B0）** | 每窗约 **8.38×10⁸ 字节** 量级的读盘（热缓存、同机一次测量）、墙钟约 **2.7–2.8 秒/窗** 量级（随系统负载略有浮动）。 |
| **协议说明** | 多窗 **同一进程连续跑**，**操作系统页缓存往往偏热**——因此 **绝对秒数** 会偏乐观；**同一对 full/prune 的比值** 仍适合说明 **相对收益**。 |

**读结果时的白话**：  
- 看到 **`ratio_bytes` 接近 0**：剪枝路径几乎 **没读** 相当于「整表那么大的数据量」——典型如 **东向窗**（数据根本不在那个经度带）。  
- 看到 **`ratio_bytes` 约 0.5～0.6**：剪枝仍 **省近一半读盘**，但窗里仍 **命中大量点**（大窗或时间段覆盖多）。  
- **`full` 一列各窗差不多**：因为 B0 **根本不看窗**，每次都把 **整 CF** 走完。

## 第五部分：我们具体改了什么——SST、Manifest、读路径（设计清单）

下面按 **存储里放什么** → **读的时候怎么用** 的顺序写，与 `docs/sst_and_manifest_plan.md`、`Record.md`、`AGENT_HANDOFF.md` 一致。

### 5.1 User key（键怎么带时空）

- 轨迹点使用带 **`0xE5` 前缀** 的 **ST user key**，可从键中解析 **时间 \(t\)** 与 **平面坐标 \((x,y)\)**（与块/文件级包围盒 **同一套不相交规则**）。
- 相关代码概念：`st_meta_user_key` 等。

### 5.2 SST 主索引：块级「时空尾巴」（Index Value 扩展）

- 在 **BlockBasedTable** 的 **主索引**里，每个索引项除了指向数据块的句柄外，还可带 **固定长度的时空摘要**（例如块内 **时间范围**、**平面 MBR** 等）。
- **作用（块级）**：迭代器在 **打开某个数据块之前**，可以用 **「块的时空包络 vs 查询窗」** 判断是否 **整块不相交**；不相交则 **跳过该块**，减少 **枚举的键** 和后续工作。
- 表构建时在开启相应表选项的前提下 **聚合** 得到这些摘要。  
- 详见计划文档中 **块级尾**、`index_value_codec`、`experimental_spatio_temporal_meta_in_index_value` 等描述。

### 5.3 SST 表属性：文件级包围盒 `st_file_bounds`

- 对每个 SST，在构建结束时还可以写入 **文件级的时空包围盒**（可理解为 **把块级信息并起来** 得到的 **整文件** \(t\)、\((x,y)\) 范围），落在表属性里（如 **`rocksdb.experimental.st_file_bounds`**）。
- **作用**：让 **不打开 SST 内部索引** 也能知道 **这个文件整体** 是否与查询窗有可能相交。

### 5.4 Manifest / FileMetaData：持久化文件级 ST 元数据

- 在 **Manifest** 记录的 **`FileMetaData`** 中增加 **可选** 的 **文件级 ST 域**（与块尾 **同构的一类结构**），通过 **`VersionEdit`** 序列化，并使用 **自定义 tag**（文档中记为 **`kExperimentalStFileBounds` / tag 17** 一类）落盘。
- **老版本二进制** 若不认识该 tag，应能 **忽略** 而不破坏兼容性（设计目标）。
- **效果**：**Version 层** 在枚举某层 SST 时，**手里有每个文件的时空包络**，为 **文件级剪枝** 提供依据。

### 5.5 读路径：`experimental_st_prune_scan`（三层协同）

在 **`SeekToFirst` + `Next` 的前向扫描** 语义下（你们 bench 用的就是这条路径），大致包含：

1. **文件级（Version / `LevelIterator`）**  
   若某 SST 的 **`st_file_meta`** 与查询窗 **轴对齐不相交**，且满足实现上的保守条件（例如 **带 range deletion 的文件不跳**，避免错误），则 **不必 `OpenTable`** 该文件——**少打开整份 SST**。

2. **块级（`BlockBasedTableIterator`）**  
   对已打开的文件，在索引上前进时，用 **索引项上的块级时空尾** 跳过 **与窗不相交的数据块**。

3. **块内键级**  
   对 **块级仍判定为相交** 的块（块内点跨度大时常见），在 **`Next`** 过程中按 **解码后的 user key 的 \((t,x,y)\)** 与查询窗比较，**不对外暴露**完全在窗外的键，从而 **大幅减少「逻辑上返回的键数」**；并为后续减少 **值读取** 铺路（具体是否已把 `bytes_read` 压到与键数同比例，取决于当前是否仍要为跳过键读 value 等实现细节）。

**注意（诚实边界）**：**`Seek(具体键)` / 反向迭代** 等路径，在你们当前设计里 **未必** 全程等价于上述剪枝语义；bench 刻意使用 **前向全表式迭代** 来对齐 **「时空范围扫描」** 类 workload。

### 5.6 Compaction / 事件时间（与 SST 形态相关）

- 另有 **按业务事件时间切分 compaction 输出** 等实验选项，影响 **SST 个数、时间是否在文件间更集中**，从而间接影响 **文件级/块级剪枝是否「够细」**。  
- 这与 **「为什么同一查询在不同库布局上 IO 不同」** 有关；**14.29M 本次** 重点是 **固定一库上的 A/B**，布局细节见 `Record.md` 与 compaction 相关脚本。

## 第六部分：为什么这次能测到「好很多」——机制直觉

用 **非形式化** 三句话总结：

1. **B0 必须把整个列族当作「与窗无关的大袋子」从头到尾翻一遍**，所以 **读盘量、时间** 对 **每个窗** 都差不多。  
2. **A 在翻之前先看「文件 / 块 / 键」的时空盒子**：盒子与查询窗 **完全不相交** 就 **不打开或不读深**，所以 **东向空窗** 可以 **近乎零读**；**大窗** 仍与很多文件/块相交，就只能 **少省一部分**，`ratio_bytes` 就会 **靠近 1 但常仍 <1**。  
3. **14.29M 与较早的 ~4.84M 子集相比**，**数据覆盖的时空范围、SST 切分、窗与数据的重叠比例** 都变了，所以 **同一标签的窗**（如 wide）对应的 **`ratio_bytes` 不会和旧实验逐点相同**——**不能简单用「键数 ×3」外推比例**；应 **以本次 TSV 为准** 写报告。

**与「设计是否有效」的关系**：  
若 **在固定实现下**，**同一库、同一窗**，**A 系统性低于 B0**（读更少、时间更短，且 **东向/微窗** 类场景 **极端悬殊**），则说明 **「多出来的 Manifest/SST/迭代器逻辑」确实在读路径上做了有效工作**，而不是纯开销。

## 第七部分：这算不算「证明了设计有效」？（非论文口径）

### 7.1 在**本次实验定义下**，可以下的结论

- **是**：在 **「无剪枝整 CF 迭代」** 这一 **明确、可复现** 的基线上，**开启剪枝 + 给定时空窗** 能 **大幅** 降低 **读字节** 与 **墙钟**（尤其 **窗与数据弱相交** 时），在 **大窗** 上仍常 **低于** 基线——这与 **文件级 / 块级 / 键级** 三层语义 **一致**。  
- **是**：**12 个设计窗 + 随机窗** 说明现象 **不是** 只依赖某一个手工 magic 窗。

### 7.2 不应过度外推的结论

- **不是**：「线上任意业务、任意访问模式都一定更快」——真实系统可能 **点查、短 range、别的索引**，不等价于 **整 CF 迭代**。  
- **不是**：「相对官方未修改 RocksDB」——对比的是 **你们 fork 内** 开关剪枝 **前后**，格式与键空间与上游 **不一定兼容**。  
- **不是**：「严格冷盘、多机、统计置信区间已完备」——当前 **热缓存、单次/少量 seed** 更适合 **工程验证 + 论文初稿**；要 **顶会级统计叙事** 需另补协议。

### 7.3 给业务方/合作方的一句话

**我们在轨迹点存进 RocksDB 的前提下，给存储层加上了「时空摘要」，让「按时间+地图矩形扫点」这种访问，相对「傻扫全表」少读盘、少花时间；在 1400 万点规模的 PKDD 子集上，用固定脚本复现了从接近 0% 到约一半读盘/时间不等的收益，与查询窗和数据的重叠程度一致——这说明改造在读路径上是「真的在干活」，而不是只改了格式没效果。**
